# Purchase Model Architecture - UNIFIED SYSTEM

## IMPORTANT: Single Model System

**⚠️ CRITICAL: DO NOT CREATE SEPARATE "PurchaseOrder" MODEL ⚠️**

This system uses a **UNIFIED PURCHASE MODEL** architecture. All purchase operations are handled by the single `Purchase.php` model.

## Architecture Decision

### Why Unified Model?

- **Single Source of Truth**: All purchase operations in one place
- **Consistent Database**: Uses `purchases` table exclusively
- **No Duplication**: Eliminates code redundancy and maintenance overhead
- **Clear Ownership**: One model, one responsibility
- **AI-Friendly**: Prevents confusion and duplicate model creation

## Model Structure

### Primary Model: `app/models/Purchase.php`

**Database Table**: `purchases` (NOT `purchase_orders`)

**Primary Methods** (Use these for new development):

- `createPurchase($data)` - Create new purchase
- `getPurchaseById($id)` - Get purchase by ID
- `updatePurchase($id, $data)` - Update purchase
- `addPurchaseItem($data)` - Add item to purchase
- `getPurchaseItems($purchaseId)` - Get purchase items
- `getPurchaseSummary()` - Get dashboard summary

**Backward Compatibility Aliases** (Legacy support only):

- `createPurchaseOrder($data)` → calls `createPurchase()`
- `getPurchaseOrderById($id)` → calls `getPurchaseById()`
- `updatePurchaseOrder($id, $data)` → calls `updatePurchase()`
- `addPurchaseOrderItem($data)` → calls `addPurchaseItem()`
- `getPurchaseOrderItems($purchaseId)` → calls `getPurchaseItems()`
- `getPurchaseOrderSummary()` → calls `getPurchaseSummary()`

## Terminology Standards

### Backend (Technical)

- **Models**: Use `Purchase` class and `purchases` table
- **Methods**: Use `purchase` naming (e.g., `getPurchase`, `createPurchase`)
- **Variables**: Use `$purchase`, `purchaseModel`
- **Database**: `purchases` table, `purchase_items` table

### Frontend (User-Facing)

- **UI Labels**: "Purchase Order" for user clarity
- **Page Titles**: "Purchase Order Management"
- **Forms**: "Purchase Order Number", "Purchase Order Details"
- **Reports**: "Purchase Order Status"

## PO Number Format

**Standard**: `PO-YYMMDD-HHMMSS`
**Example**: `PO-250810-143052`
**Generated by**: `Purchase::generatePONumber()`

## Status Workflow

**Statuses**: pending → sent → shipped → received → completed
**Cancellation**: Any status can go to `cancelled`
**Managed by**: `Purchase::updateStatus()` method
